# 教學動態白板 — 修訂版開發計畫 (v2)

> 本文件為給工程開發(Cursor / Claude Code)使用的技術規格與實作藍圖。
> 與 v1 相比,核心修正為三點:**(1) 改採兩層 DSL 架構**、**(2) 座標排版交給排版引擎**、**(3) MVP 先砍掉語音,先驗證「文字 → 板書 → 筆記」**。

---

## 0. 一句話定位

把講課內容轉成「結構化、極簡的板書視覺」(文字框、箭頭、簡單圖形),並在課後自動輸出乾淨筆記。**核心價值不在炫,而在「穩定生成好看且正確的板書結構」。**

---

## 1. 與 v1 的關鍵架構修正

### 修正 A:不要讓 LLM 直接輸出完整 Excalidraw JSON

Excalidraw 每個元素需要 `id`、`seed`、`versionNonce`、`roughness`、`strokeColor`、`fillStyle` 等十幾個欄位。讓 LLM 直接吐完整規格非常脆弱,常漏欄位或產生不合法結構。

**改為兩層:**

1. **語意層(LLM 負責)**:LLM 只輸出精簡、好寫的中介 DSL(描述「有哪些節點、誰連到誰、分組關係」)。
2. **渲染層(確定性程式碼負責)**:由純函式把 DSL 轉成合法的 Excalidraw 元素並填滿所有欄位。

好處:LLM 輸出短、合法率高、token 成本低、可單元測試。

### 修正 B:座標 / 排版不交給 LLM 算

LLM 算 x/y 與避免重疊的能力很差,超過 4–5 個元素就會開始重疊跳動。**LLM 只描述語意關係,實際座標交給排版引擎**(流程圖類用 `dagre` 或 `elk`)。這同時解掉「元件重疊、畫面跳動」。

### 修正 C:MVP 先砍語音

最不確定的風險是「文字稿 → 穩定好看的板書」。這段若做不出來,即時化沒有意義;若做得好,即時化只是工程問題。
**MVP 範圍 = 貼逐字稿 / 打字 → 生成板書 → 匯出筆記**。語音是後續疊加的「輸入方式」,不是要先攻克的核心。

## 2. 系統架構

```
┌─────────────┐   text    ┌──────────────┐   DSL(JSON)   ┌────────────────┐
│ 輸入層       │ ────────▶ │ LLM 推論層    │ ────────────▶ │ DSL 驗證 + 轉換 │
│ (打字/貼稿/   │           │ (語意抽取)    │   (Zod 驗證)   │ (layout + 渲染) │
│  之後接語音)  │           └──────────────┘               └────────┬───────┘
└─────────────┘                  ▲                                  │ Excalidraw elements
                                 │ 板上現況摘要(state context)        ▼
                          ┌──────┴───────┐                  ┌────────────────┐
                          │ 白板狀態管理   │ ◀──────────────  │ Excalidraw 畫布 │
                          │ (node registry)│   teacher edits  │ (React Island) │
                          └──────────────┘                  └────────────────┘
                                                                     │ 課後
                                                                     ▼
                                                            ┌────────────────┐
                                                            │ 匯出 + 摘要筆記 │
                                                            └────────────────┘
```

**資料流的核心原則:** 板書是「累積」而非「重畫」。每次 LLM 推論都要餵入「板上現況摘要」,讓它輸出的是 `add` / `connect` 動作,而不是整張重來。

## 3. 技術堆疊

| 層 | 選型 | 備註 |
|---|---|---|
| 前端框架 | Astro + Tailwind CSS | 互動元件用 React Islands |
| 白板核心 | `@excalidraw/excalidraw` | 以 `client:only="react"` 渲染 |
| DSL 驗證 | `zod` | 強制 LLM 輸出符合 schema |
| 排版引擎 | `dagre`(MVP)/ 之後可換 `elkjs` | 算節點座標與連線走向 |
| LLM | GPT-4o 或 Gemini 1.5 Pro | 需強 JSON 輸出;用 structured output / function calling |
| STT(階段 4 才接) | OpenAI Whisper / 低延遲串流服務 | Web Audio API 擷取麥克風 |

## 4. 中介 DSL 規格(本專案的核心契約)

LLM 的唯一輸出格式。刻意精簡,**不含任何座標**。

```ts
// dsl.ts
import { z } from "zod";

export const NodeSchema = z.object({
  id: z.string(),                         // 穩定唯一 id,LLM 自行命名(如 "sun", "plant")
  label: z.string(),                      // 顯示文字
  shape: z.enum(["circle", "rect", "diamond", "text", "ellipse"]),
  group: z.string().optional(),           // 同組節點會被排版引擎聚在一起
  emphasis: z.enum(["normal", "strong"]).default("normal"),
});

export const EdgeSchema = z.object({
  from: z.string(),                       // node id
  to: z.string(),                         // node id
  label: z.string().optional(),           // 箭頭上的文字
  style: z.enum(["arrow", "line"]).default("arrow"),
});

export const BoardOpSchema = z.object({
  op: z.enum(["add", "connect", "update", "remove"]),
  nodes: z.array(NodeSchema).default([]),
  edges: z.array(EdgeSchema).default([]),
  removeIds: z.array(z.string()).default([]),
});

export const LLMOutputSchema = z.object({
  ops: z.array(BoardOpSchema),
  // 給課後筆記用的純文字重點(可空)
  noteHint: z.string().optional(),
});

export type LLMOutput = z.infer<typeof LLMOutputSchema>;
```

**設計重點**
- `id` 必須穩定:同一概念再次被提到時,LLM 要沿用既有 id 才能正確 `connect` / `update`。
- 沒有座標:座標一律由排版引擎在渲染層計算。
- `op` 化:支援增量更新與老師編輯後的同步。

## 5. 白板狀態管理(node registry)

維護一份「目前板上有什麼」的真實來源,用於兩件事:餵 context 給 LLM、處理老師編輯。

```ts
type BoardState = {
  nodes: Map<string, NodeSchema & { locked: boolean }>;
  edges: EdgeSchema[];
};
```

- 每次要呼叫 LLM 前,把 `BoardState` 壓縮成一段精簡摘要(id + label + group 即可)塞進 prompt。
- **老師編輯衝突的 MVP 處理**:被老師手動移動 / 編輯過的節點標記 `locked = true`,LLM 不得 `update` / `remove` 鎖定節點(在轉換層擋掉),避免雙向不同步。先求簡單可用。

## 6. 渲染層:DSL → Excalidraw

純函式,**可完整單元測試**,不碰網路。

```ts
// render.ts
import dagre from "dagre";

export function dslToExcalidraw(state: BoardState): ExcalidrawElement[] {
  // 1. 用 dagre 建圖,餵入 nodes / edges,算出每個 node 的 x,y
  // 2. 依 shape 對應到 Excalidraw 元素型別,補齊所有必填欄位
  //    (id, seed, versionNonce, roughness, strokeColor, ...)
  // 3. emphasis=strong → 加粗 strokeWidth / 換色
  // 4. edges → arrow 元素,bind 到 from/to 的 elementId
  // 回傳合法 elements,交給 excalidrawAPI.updateScene({ elements })
}
```

驗收:給定一組固定 DSL,輸出的 elements 丟進 Excalidraw 不報錯、無重疊、連線正確綁定。

## 7. 分階段實作計畫

### 階段一:前端基礎 + 白板嵌入 + 渲染層
**目標:** 不靠 AI,先讓「DSL → 畫面」整條渲染管線穩定。

1. 初始化 Astro + Tailwind,建立極簡無干擾版型。
2. 以 `client:only="react"` 嵌入 Excalidraw,取得 `excalidrawAPI` ref。
3. 實作 `dsl.ts`(Zod schema)與 `render.ts`(dagre + 轉換)。
4. **PoC:** 手寫 2–3 組 DSL 測試資料,呼叫 `updateScene`,確認圖形 / 文字 / 箭頭正確長出且不重疊。

✅ 完成標準:寫死的 DSL 能穩定畫出乾淨板書,排版引擎可避免重疊。

---

### 階段二:AI 邏輯(文字 → DSL)
**目標:** 打通「自然語言 → 合法 DSL」,這是專案成敗關鍵。

1. **Prompt 設計:** 系統指令要求 LLM 只輸出符合第 4 節 schema 的 JSON;優先使用模型的 structured output / function calling 強制格式。
2. **驗證閘門:** LLM 輸出一律過 `LLMOutputSchema.parse()`;失敗則重試或丟棄,絕不直接送進渲染層。
3. **評測集(重要):** 準備 8–10 段教學文字稿 + 人工標註的「期望節點 / 連線結構」,當回歸測試。例:「左邊畫圓代表太陽,右邊方塊代表植物,中間箭頭連接」→ 期望 2 nodes + 1 edge。
4. **預載機制:** 一個簡單資料結構,課前讀入專有名詞清單 / 底圖模版,注入 prompt 提升準確度。

✅ 完成標準:評測集的結構正確率達可接受門檻(先抓 ≥ 80%),且 100% 輸出皆通過 Zod 驗證。

---

### 階段三:語音串接與即時渲染
**目標:** 把語音接成「輸入方式」,並調好節奏。

1. **語音擷取:** Web Audio API 取麥克風 → 切片送 STT → 純文字。
2. **管線串接:** 文字 → LLM → DSL → 渲染。
3. **節奏設計(關鍵):** **不追求逐句即時,改以「語意段落」為單位浮現**,更像真實老師寫板書,也讓延遲變成設計選擇而非 bug。
   - 加入 **debounce / 靜默偵測**:講者停頓 N 秒才觸發一次推論。
   - 控制 API 呼叫頻率以壓成本與避免畫面抖動。

✅ 完成標準:連續講述時,板書以自然節奏分塊浮現,無明顯重疊 / 跳動。

---

### 階段四:課後筆記與輸出
**目標:** 產出實用學習資產。

1. **畫面匯出:** 用 Excalidraw 內建 API 匯出唯讀 PNG / SVG。
2. **自動摘要:** 把逐字稿 + 最終 DSL 結構交給 LLM,輸出條理分明的文字版重點筆記(可結合各 op 的 `noteHint`)。
3. 打包:板書圖 + 文字筆記,供課後分發。

✅ 完成標準:一鍵產出「板書圖 + 重點筆記」。

---

## 8. 橫向風險與須留意事項

- **延遲鏈會疊加**(STT 1–3s + LLM 1–3s),這是「跟上講者」願景的最大風險 → 用階段三的「語意段落浮現」吸收。
- **成本:** 持續 STT + 高頻 LLM 呼叫,一小時課程成本需先估算,會影響商業模式 → 用 debounce 降頻、評估較便宜模型跑語意抽取。
- **狀態同步:** 累積式板書 + 老師編輯是隱藏工程量 → MVP 用「鎖定節點」簡化處理。
- **id 穩定性:** LLM 重複命名或亂改 id 會破壞增量更新 → 在 prompt 強調沿用既有 id,並在轉換層做容錯比對。

---

## 9. 建議的開發起手式(給 Claude Code)

依序執行,每步都有明確驗收,適合逐步開發:

1. `scaffold`:Astro + Tailwind + Excalidraw island,畫面能跑。
2. `dsl.ts` + `render.ts` + 單元測試(寫死 DSL → 合法 elements)。
3. 在頁面接一個「貼文字稿」輸入框 →(先 mock LLM,回固定 DSL)→ 跑通整條渲染。
4. 接真 LLM + Zod 驗證閘門 + 評測集。
5. 匯出 + 摘要(階段四)。
6. 最後才接語音(階段三)。

> 建議先把第 1–4 步做成可 demo 的「文字版 MVP」,再決定是否投入語音。
